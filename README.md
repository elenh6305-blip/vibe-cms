# Vibe CMS

**一句话版本**：这个 skill 让 AI 编程助手自动把项目里所有用户能看到的文字抓出来，生成一个 Markdown 表格文件。你可以直接在表格里改硬编码的文案，改完之后 AI 自动把变更回填。

---

## 这个 Skill 要解决什么问题？

你用 AI 对话搭出了一个 App 。界面好看，功能能跑。但是现在你想改一下首页标题的措辞，或者把按钮上的"开始"换成"出发"。

你会怎么做？跟 AI 说"帮我把首页那个标题改一下"？

问题是——你的项目可能有几十个文件，上百句用户能看到的文字，散落在各个角落。你不确定"首页标题"在 AI 眼里是哪个文件的哪一行，AI 也可能改错地方，或者漏改了某处。你越改越心虚，因为你**看不到全貌**。

这就是 Vibe CMS 要解决的问题：**让你看到自己 App 里的每一句文字，并且能精确地改它们。**

AI 会扫描你的整个项目，把所有用户可见的文字——按钮、标题、错误提示、placeholder、AI 提示词——全部提取出来，整理成一张清清楚楚的 Markdown 表格：

| 类型     | 位置                         | 当前文案             | 建议新文案 |
| -------- | ---------------------------- | -------------------- | ---------- |
| 首页标题 | `src/pages/Home.jsx:47`      | `今天，从一步开始。` |            |
| 提交按钮 | `src/components/Form.jsx:49` | `开始部署`           |            |

你可以把生成好的 Markdown 文件发给非技术人员（当然也可以自己用），他们能像填 Excel 一样直接编辑，也能直接修改 Prompt ，完全不需要碰代码。你填好想要的新内容后，告诉 AI "按 CMS 文档回填代码"，AI 就会逐条精确替换。不用你去翻代码，也不怕改错。

```
你/非技术人员  ——编辑表格——>  Markdown 文件 (CMS)  ——AI回填——>  源代码
```

没有数据库，没有后端服务，没有管理面板。一个 `.md` 文件就是你的"CMS"。
它能够自动修正错误格式，并扫描填写过程可能出现的人为失误。

---

## 两个阶段 + 一个功能

### Phase A：扫描 → 生成文档

你说"帮我整理所有前端文案"，AI 会：

1. 遍历项目源码（`src/`、`index.html`、`prompts.js` 等）
2. 找出所有硬编码的用户可见文字——按钮文案、页面标题、错误提示、placeholder、aria-label、AI 提示词……所有的
3. 按页面/功能区分组，生成一个 Markdown 文件（默认存到 `docs/cms-copy-inventory.md`）

生成的文档长这样（截取一小段）：

| 类型     | 位置                                | 当前文案               | 建议新文案 |
| -------- | ----------------------------------- | ---------------------- | ---------- |
| 首页标题 | `src/pages/HomePage.jsx:47`         | `打造你的应用。`       |            |
| 计数提示 | `src/components/SearchInput.jsx:49` | `已完成 {length} 项！` |            |

重点是最后一列 **"建议新文案"** 留空了。这一列就是给编辑者用的。

AI 提示词（和多行文本）用代码块展示，而不是塞进表格：

当前内容：
~~~~text
你是一位前端程序员。
一步一步完成以下任务……
~~~~

建议新版提示词：
（这里留空，等编辑者填写）
~~~~

### Phase B：回填 → 应用变更

编辑者改完文档后，你说"按 CMS 文档回填代码"，AI 会：

1. 读取 CMS 文档，找出所有"建议新文案"非空的行
2. 逐条去源码里核实——打开对应文件、定位到那一行（±3 行容错）、确认"当前文案"确实存在
3. 确认无误后执行替换 🚦 **双重防呆检测 (Dry-run) **：
   - **变量防呆**：如果原代码是 `已完成 {count} 项`，而你填的文案忘记带上变量 `{count}`，AI 会拒绝替换该行以防止页面崩溃。
   - **语法防呆**：结合源码实际使用的字符串定界符来检查。比如单引号字符串里要防未转义的 `'`，双引号字符串里要防未转义的 `"`，避免把字符串意外截断导致打包失败。
4. 输出一份变更报告：成功几处、跳过几处、有没有匹配不上的

如果某条文案在代码里找不到（比如行号漂移了太多），AI 会跳过并告诉你，不会猜测性地乱替换。

## Phase C：增量同步升级 (Incremental Update)

你可能喜欢边改边做，或者和他人协作编辑。有时你的 CMS文档 才修改到一半，代码里有了新的文案和组件。这个 Skill 考虑到了这种情况，你不需要推倒原有的文档。

你只需要说"我新增了页面，帮我把新文案**增量同步**到清单里"，就会自动触发**Phase C**，skill会重扫项目，**抓取新增词条并自动追加到现有的文档末尾，而且不会覆盖**你之前填入文档但没有应用的修改内容。

## 怎么安装

推荐直接用 `npx` 安装：

```bash
npx skills add elenh6305-blip/vibe-cms --skill vibe-cms
```

安装完成后，重启你的 AI 编程工具以加载这个 skill。

## 怎么用 (包含自动引导，也可以使用触发口令)

**生成文档**（以下任意一句都行）：

- "帮我整理所有前端文案"
- "生成一份内容清单"
- "list all the text in my app"

**回填新修改**：
- "按 CMS 文档回填代码"
- "apply the CMS document"
- "把文案改动应用到代码里"

**增量同步**：
- "新增了一个设置面板，帮我增量同步一下 CMS 文档"

## 适用范围和局限

**适用**：任何前端框架（React、Vue、Svelte、Next.js、纯 HTML……）中硬编码在源文件里的静态文本。

**不适用**：
- 已经用了 i18n 库（如 react-i18next）的项目——那些文案已经有自己的管理方式了
- 来自数据库、API 或用户输入的动态内容——CMS 文档会标注这些内容的存在，但不会尝试管理它们
- 行号会过时——这是设计上的已知问题。CMS 里的 `[文件路径:行号]` 本质是个快照。即便你后续大改代码导致行数漂移，回填逻辑主要依赖"原文字字符串匹配"和防呆，行号只是为了提高容错，安全性依然极高。如果行号偏离太多没找到，触发一次 Phase C（增量同步）就会重刷所有行号。
